El Concepto "2.0"

Pizza as a Service es una analogía famosa para explicar la gestión de TI. La versión 2.0 actualiza este modelo incorporando arquitecturas modernas como Contenedores (CaaS) y Serverless (FaaS).

En lugar de cocinar desde cero (On-Premises), la nube nos permite abstraer capas de responsabilidad operativa.

IaaS (Infraestructura)

Descripción: "Take and Bake". El proveedor de nube te da los componentes básicos (Redes, Almacenamiento, Servidores virtuales). Tú debes instalar el Sistema Operativo, middleware y la aplicación.

IaaS: El Rol del Hipervisor

  • El proveedor opera un hipervisor (ej. KVM, Xen, Hyper-V) que abstrae el hardware físico y lo convierte en múltiples máquinas virtuales aisladas.
  • Cada VM tiene su propia CPU virtual, memoria RAM y disco asignados. El usuario ve un servidor completo, pero comparte el hierro físico con otros inquilinos de forma transparente y segura.
  • El usuario toma control desde el sistema operativo hacia arriba; todo lo que está por debajo es responsabilidad del proveedor.

IaaS: Pros y Contras

  • Ventajas: Control total sobre el Sistema Operativo, gestión de firewalls y redes internas (VPCs), ideal para sistemas legacy, alta flexibilidad de configuración.
  • Desventajas: Requiere un equipo de TI capacitado para administrar parches de vulnerabilidades del kernel, seguridad del SO, escalamiento manual y gestión de respaldos a nivel de disco.

IaaS: Casos y Ejemplos

  • Lift-and-Shift (Rehost): Mover una VM on-premises a la nube sin refactorizar el código. Es la estrategia más rápida de migración, ideal cuando el sistema no puede ser re-arquitectado pero sí necesita dejar el datacenter físico.
  • Bases de Datos de Alto Rendimiento: Cargas de trabajo que requieren IOPS garantizados (operaciones de E/S por segundo) o configuraciones a nivel de kernel que un PaaS gestionado no permitiría (ej. tuning de parámetros de PostgreSQL o Oracle RAC).
  • Ejemplos: Amazon EC2 (AWS), Google Compute Engine (GCP), Azure Virtual Machines.

PaaS (Plataforma)

Descripción: "Pizza Delivery". El proveedor maneja todo el entorno subyacente (Servidor, SO, Runtime). El equipo de desarrollo solo se enfoca en subir el código (la receta) y manejar los datos.

PaaS: Runtime y Ciclo de Despliegue

  • El proveedor expone un Runtime gestionado (Node.js, Python, .NET Core, Java) y se encarga de su actualización y parcheo de seguridad.
  • El despliegue se apoya en buildpacks o pipelines integrados: el desarrollador hace git push y la plataforma detecta el lenguaje, instala dependencias, compila y levanta la aplicación automáticamente.
  • El ciclo completo —build, release, run— sigue el estándar 12-Factor App, lo que garantiza portabilidad y reproducibilidad entre entornos.

PaaS: Pros y Contras

  • Ventajas: Despliegue de aplicaciones ultrarrápido, auto-escalado integrado, cero mantenimiento de servidores.
  • Desventajas: Vendor Lock-in profundo al depender de add-ons propietarios (ej. Heroku Postgres, AWS Elastic Cache); además, no es posible instalar dependencias binarias a nivel de SO (ej. librerías C nativas o drivers personalizados), lo que bloquea ciertos casos de uso de ciencia de datos o procesamiento de señales.

PaaS: Casos y Ejemplos

  • Entornos de Staging Rápidos: Equipos pequeños que necesitan levantar un entorno de pruebas idéntico a producción en minutos, sin involucrar a un equipo de operaciones.
  • Microservicios Estándar: Arquitecturas donde el time-to-market es crítico y cada servicio usa un stack convencional (REST API en Node, worker en Python). PaaS elimina la fricción operativa para que el equipo se enfoque en la lógica de negocio.
  • Ejemplos: Heroku, AWS Elastic Beanstalk, Google App Engine.

FaaS (Serverless)

Descripción: "Pizza por Rebanada". Es el verdadero modelo 2.0. No hay servidores visibles ni encendidos todo el tiempo. Subes un fragmento de código (función) que se ejecuta solo cuando ocurre un evento específico.

FaaS: Arquitectura Orientada a Eventos

  • El paradigma central es Event-Driven: la función "duerme" empaquetada en un micro-contenedor frío hasta que un trigger la despierta.
  • Los triggers más comunes son: una petición HTTP (API Gateway), un cambio en base de datos (DynamoDB Streams, Firestore triggers) o un mensaje en una cola (SQS, Pub/Sub, Kafka).
  • Una vez procesado el evento, el contenedor puede mantenerse "tibio" brevemente para reutilización, o apagarse — el proveedor decide. El desarrollador nunca gestiona servidores ni contenedores directamente.

FaaS: Pros y Contras

  • Ventajas: Pago estricto por uso (cobro por tiempo de ejecución, con resolución de milisegundos), alta escalabilidad automática con límites de concurrencia configurables por cuenta y región.
  • Desventajas: Cold Start: cuando la función lleva tiempo inactiva, el proveedor debe aprovisionar y arrancar un nuevo micro-contenedor, añadiendo latencia (típicamente 100ms–2s). Además, la trazabilidad distribuida es compleja: sin herramientas como OpenTelemetry o AWS CloudWatch X-Ray, depurar una cadena de funciones asíncronas es muy difícil.

FaaS: Casos y Ejemplos

  • IoT y Telemetría: Procesamiento de ráfagas de datos de sensores: miles de dispositivos pueden enviar lecturas simultáneamente; FaaS escala horizontalmente sin intervención y descarta recursos cuando el flujo cae.
  • Webhooks de Pago: Eventos de alta concurrencia y baja duración (ej. confirmaciones de Stripe o PayPal). Cada notificación dispara una función independiente, garantizando aislamiento y sin riesgo de saturar un servidor compartido.
  • Ejemplos: AWS Lambda, Google Cloud Functions, Azure Functions.

SaaS (Software)

Descripción: "Dining Out" (Comer en restaurante). Consumes el software como un producto final a través de internet. El proveedor maneja absolutamente toda la infraestructura y el código.

SaaS: Multi-tenancy

  • La mayoría de los SaaS operan bajo un modelo multi-inquilino (Multi-tenancy): múltiples clientes comparten la misma instancia de la aplicación e infraestructura subyacente.
  • El aislamiento de datos es lógico, no físico: cada cliente ve únicamente sus datos gracias a identificadores de tenant (ej. tenant_id en cada query), pero las tablas pueden coexistir en la misma base de datos.
  • Esto reduce el costo operativo del proveedor pero introduce riesgos de cross-tenant data leakage si los controles de acceso no están bien implementados.

SaaS: Pros y Contras

  • Ventajas: Cero mantenimiento para el usuario, acceso universal desde cualquier navegador, esquema de costos predecible (suscripción).
  • Desventajas: Shadow IT: usuarios o áreas del negocio contratan software SaaS sin supervisión del equipo de TI, generando riesgos de seguridad y fragmentación de datos. Además, las restricciones de API rate-limiting dificultan la extracción masiva de datos hacia almacenes locales (data warehouses), comprometiendo las capacidades analíticas propias.

SaaS: Casos y Ejemplos

  • Casos de Uso: Herramientas de productividad corporativa, correo electrónico, CRMs para ventas y colaboración en equipo.
  • Ejemplos: Google Workspace (Gmail/Drive), Salesforce, Slack, Microsoft 365.
